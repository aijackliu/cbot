## TL;DR

| 判斷 | 內容 |
|------|------|
| **不是「SaaS 消失」** | 消失的更可能是**人類必須登入後台 UI 的使用方式** |
| **Headless** | 不是拿掉軟體，而是把 **GUI 從產品核心挪開**；價值在資料、邏輯、權限、例外規則 |
| **Agent 三種事** | ① 查詢 ② 做事（寫入／冒充身份／席位） ③ 分析（多系統、需驗證） |
| **SAP 為什麼還在** | 封裝的是**公司怎麼運作**，不是「Postgres + API」能取代的表格 |
| **整場遊戲** | **例外處理** + **context（未寫進欄位的政策）** + 時間累積 |
| **新創機會** | 不正面對決 SoR；卡在巨頭之間、跨部門翻譯層、資料之上**採取行動**、實體垂直資料 |

**一句話：** SaaS 可能從「給人點的軟體」變成「給 Agent 存取與執行的基礎設施」；UI 退場，**系統記錄與商業邏輯不會退場**。

---

## ① Headless Software 是什麼

- 傳統軟體：**為人類存取**而建（工作流、畫面、肌肉記憶）。
- Agent 世界：多半走 **API / MCP / 聊天／CLI**，不在乎按鈕與 Dashboard。
- Salesforce **Headless 360**：功能層（客戶、商機、流程）與 GUI **解耦** → 開放 API、MCP、CLI。
- Seema 觀察：Salesforce 個案多半是 **既有 API 的品牌再包裝**；真正有意義的是巨頭在為 **agent 存取系統記錄** 重新定位。
- Notion headless 更「合理」：使用者技術能力較高、更習慣 agentic 建構。
- 類比：Slackbot 用量暴增 → 人不必進 Salesforce UI 也能查 CRM。

Sinofsky：**定義地獄**——新浪潮會給舊東西貼新詞；「agent」某種程度上是「跑很久、可能跑不完的程式」的行銷包裝。

### Agent 做的三類事（Sinofsky）

| 類型 | 本質 | 難點 |
|------|------|------|
| **Look up** | 查詢 | 輕量；很多 headless 只是換介面查詢 |
| **Do something** | 寫入／執行動作 | 身份、憑證、席位、權限、企業政策 |
| **Analyze** | 跨系統分析 | 運算與迭代適合 agent；**幻覺與逐步驗證**最重 |

Headless ≠ Agent；要先講清楚自己在哪一種。

---

## ② 什麼讓企業軟體黏著

| 來源 | 機制 |
|------|------|
| 人類互動 | 讀寫頻率、下游流程、未成文 SOP、肌肉記憶 |
| 組織政治 | 新業務副總「指定 Salesforce」；財務／行銷依賴同一真相來源 |
| 合規稽核 | 必須乾淨追蹤；替代方案少 |
| **收錢** | 已在付費 → 停付極難（Sinofsky） |
| 怪細節 | 例：Outlook 的 **代理存取、共用行事曆例外**——沒人刻意設計成護城河，卻讓 60 萬席位拔不掉 |

**慣性（inertia）** 與「已付費很久、滲進日常」比功能清單更重要。

---

## ③ 為什麼 SAP（與同類 SoR）可能不會消失

| 迷思 | 反駁 |
|------|------|
| Postgres + API = 取代 SAP | 資料庫不懂公司如何運作 |
| 40 人新創 OCR 報帳 = 企業報帳 | 1 萬人、20 國、法律與差旅政策 + 內部政策 → **編碼成這家公司獨有** |
| vibe code 一個 CRM | 難的是：記什麼、組織邊界、誰長期維護、隨業務演進 |

- SAP／沃爾瑪級系統：**拿掉 ≈ 公司定義被拿掉**（規則在系統裡，不只在軟體授權裡）。
- Larry Ellison 當年罵企業客製化、鼓吹「用 80% 標準方案」——企業界反彈：現實不是 80% 夠用。
- 車廠差異往往在 **營運與 ERP 裡的決策流程**，不在「都會造車」。
- Excel／高盛故事：差異化在 **怎麼用工具**（外掛、流程），不是軟體本身。
- 周邊新創：在 SAP **之上／旁邊** 做易用層（對話式查詢、報表），**不連根拔起**。
- 企業軟體幾乎「什麼報表都能出」——使用者不知道怎麼叫出來；逃生閥常是 **匯出 Excel／CSV／PDF**。LLM 讓匯出後 ad hoc 分析變強。
- CRM 史：Excel → Siebel → Salesforce；Salesforce 也是業務 **強制蒐集欄位** 的機制。
- Agent 陌生開發仍需：**資料存取 + context**（例外、地區話術、未寫進欄位的政策）= context graph 的實質。

---

## ④ 例外處理才是整場遊戲

- 企業裡「有趣的事」幾乎都是 **例外**（定價、客戶個案、逾期、授權算錯）。
- 例外多半 **沒被系統性記錄**；語音 agent／電腦操作 agent 可觀察人類點擊與回應，開始累積情境——但要 **跨完整銷售週期**，不是三天蒐完。
- 工程師易把 headless =「商業流程寫成 API」；**非工程師往往說不清自己怎麼解客戶問題**。
- 亞馬遜：優先站客戶一邊（錯貨直接補寄）→ 用資料改善倉儲／描述；定義「例外」的方式在變。
- 權限：誰讀寫、何時、agent 之間如何互動——可解，但要時間。
- 理想單一真相來源 vs 多方寫入 → 新治理問題。

---

## ⑤ 生產力創造新場景，不是消滅工作

- 指數成長當下難理解；勿把指數當線性外推。
- 自動化「輕鬆層」之後，人們會發明 **新工作**（商務差旅分析、遠端工作優化…）。
- 長尾不會變短，只是 **換方式變長**。
- 法律：合約更快 → 可能更長、更複雜、更多爭議生態。
- 放射科都市傳說：愛用 AI 卻人力短缺 → 需求非靜態。
- 永遠有 **人與人成交、出差、現場** → 持續有 data exhaust 要捕捉（語音、郵件、文件）。
- 「完成／結帳／認列」像開源 release：**需要敘事與責任軌跡**，不是一支 API 就結束。
- 專業知識在 **未結構化文件雲**（Box／Word／Excel）；AI 第一次真能量化挖這些。

---

## ⑥ MCP 與中介層：歷史重演

- MCP 熱潮 ↔ 當年微軟反壟斷論點中的 **middleware** 想像。
- 工程師想要乾淨 CLI／API；**沒有供應商甘願只當啞資料庫**。
- 客戶也不想拼湊一堆最弱環節決定穩定性的供應商堆疊。
- Workday／Salesforce：API 在，但 **完整乾淨抽乾** 通常不行——沒有誘因把自己做成 dumb DB。
- Seema 三條路：
  1. 用巨頭當後端 + 其 agent（巨頭未必甘當背景；也不看好巨頭 agent 品質）
  2. **DIY 重建 SoR**（對 Fortune 500 = 活體換心）
  3. **與 SAP 等並肩／疊加**，同時用語音、錄音、文件攝入 **長出新系統記錄**（a16z 投資主軸）

## ⑦ 新創機會（收斂）

| 方向 | 說明 |
|------|------|
| 資料之上**採取行動** | 不只記通話 → 優先名單、流失預警、主動 outreach |
| Agentic 閉環 | 發訊→看回覆→學有效策略；地區話術基準資料 |
| 實體垂直 | 建築、製造：現場人機資料 + 既有軟體資料 |
| **不正面對決品類** | 卡在兩個把 AI 硬掛邊上的巨頭 **中間**；用新方式做事（HTTP/HTML 取代 client-server 心智，而非功能清單對打） |
| **跨部門翻譯層** | 軟體過去賣單一部門；部門交界（帳單、交易、情境）是空白 |
| **公司內部網路效應** | 外部 B2B 網路效應因合規難；內部 chat／跨職能工具可病毒擴散（Excel 電梯廣告 ↔ 今日 chat） |
| 連結不溝通的部門 | 類 Figma 連設計與產品；IT×財務預測等 |

---

## 可帶走的決策句

1. 評估 SaaS：問 **SoR 裡編碼了哪些規則／例外／合規**，不是只問 UI 好不好用。  
2. 做 Agent：先分清 **lookup / do / analyze**，權限與驗證策略不同。  
3. 做新創：避免「vibe 一個 CRM 取代 Salesforce」；找 **巨頭之間、流程之間、易用層 + 新資料排放**。  
4. Headless 行銷 vs 真實：很多是 API 再包裝；看 **是否願意被上層架空** 與 **寫入是否真開放**。  
5. 護城河：收錢、慣性、怪細節、編碼進系統的外部力量（法規／公司本身）。
