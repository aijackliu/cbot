## 0. 一頁定位（可貼牆）

```
定位：繁中「現場 AI 作業系統」
主航道：理解｜記憶｜執行
收費：成功任務額度 + B2B 情境包
護城河：場景深度 × Agent 路由 × 歷程計費 × 信任設計
不做：功能數競賽、假比價 headline、全時監控敘事
```

| 維度 | 定義 |
|------|------|
| **我們是誰** | 用眼鏡「看／聽現場」，用 Agent 辦事，用可計量歷程證明價值並收費的現場 AI OS |
| **我們不是誰** | 不是 Meta 時尚硬體；不是功能超市 App；不是全時監視裝置 |
| **對消費者一句** | 戴上就能聽懂菜單、記住會議、問現場問題——雙手空著，答案在耳邊 |
| **對 B2B 一句** | 可白標的眼鏡 Agent 堆疊：意圖路由、視覺工具、稽核計費 |
| **地理優勢** | 繁中語境、亞洲生活場景（菜單、外食、會議、旅行） |

### 不做清單（硬規則）

1. 不宣傳「真比價」直到外部價 API 上線且可驗  
2. 不做 silently always-on 錄影／錄音敘事  
3. 不以星座／運勢作為品牌主軸（可作 Lab 低成本填充）  
4. 不與 Meta 比出貨量與時尚渠道

## 1. 問題與機會

### 1.1 用戶問題

| 現場 | 痛點 | 手機 AI 為何不夠 |
|------|------|------------------|
| 外食／旅行 | 菜單看不懂、要邊拍邊查 | 雙手被佔、切 App 慢 |
| 會議／拜訪 | 要記重點又要對談 | 打字斷互動 |
| 移動中 | 想問天氣／設行程／播音樂 | 掏手機成本高 |

### 1.2 商業機會

- 眼鏡硬體出貨起量；功能訂閱已被市場驗證（額度制可行）  
- 差異化不在硬體軍備，在 **繁中場景深度 + Agent 執行 + 可計費歷程**  
- B2C 訂閱額度與 B2B 情境包可並行  

### 1.3 內部約束（必須寫進程式與文案）

| 約束 | 產品含義 |
|------|----------|
| 相機僅能實體鍵觸發（SDK） | UI 統一：「請按下眼鏡實體拍照鍵」；轉譯為「主動才拍」信任賣點 |
| WebRTC 語音鏈未定 | 語音金路徑需備援；不承諾「全時語音 Agent」直到對齊 |
| LLM 成本偏高 | 每個 Core 任務有 LLM 次數 SLA；模板 TTS／日批／cache 為必做 |
| 歷程 feature 命名粗糙 | 計費與產品分析前必須正規化 `feature_key` |

## 3. 三航道總覽

```
              現場 AI 作業系統
         ┌────────────┬────────────┬────────────┐
         │  理解 Pack  │  記憶 Pack  │  執行 Pack  │
         │ See→Understand│ Hear→Remember│ Intent→Action│
         └────────────┴────────────┴────────────┘
                         │
              歷程 · 計費 · 隱私 · 成本控制（平台層）
```

| 航道 | 一句話 | Core 能力 | 主要 API／模組 | 收費角色 |
|------|--------|-----------|----------------|----------|
| **理解** | 看一眼，聽懂現場 | 菜單翻譯、文件 OCR、（可選）營養 | `/api/visual/menu-translate`、`/doc-scan`、`/nutrition` | Vision 次數額度 |
| **記憶** | 聽過的，變成可回看 | 會議錄音摘要、行事曆寫入 | `/api/speech/transcribe`、calendar action | 會議分鐘／摘要次數 |
| **執行** | 說一句，辦一件 | 意圖路由、天氣短答、喝水／簡單記錄、音樂 | `/api/intents/parse`、weather、hydration、music actions | 基礎方案含；進階工具另計 |

### 功能分層（治理）

| 層 | 資源 | 成員 | 規則 |
|----|------|------|------|
| **Core** | 70% | 上表三航道金路徑 | 進 KPI、進訂閱、進對外 headline |
| **Support** | 20% | 天氣、喝水、音樂、裝置連線 | 服務日活，可降級 |
| **Lab** | 10% | 星座、植物、景點、人物 Wiki、商品辨識（無真價） | 不進 headline；可關；不佔成本預算優先 |

## 4. 航道一｜理解（See → Understand）

### 4.1 目標與非目標

| 目標 | 非目標 |
|------|--------|
| 外食／旅行時，免手持手機完成「看→懂→聽」 | 不承諾全語系、全菜系完美翻譯 |
| 文件重點可聽、可對照 | 不做完整 PDF 編輯器（匯出可 Phase B） |
| 營養為可選加值 | 不把「醫療級營養建議」當宣稱 |

### 4.2 用戶故事

1. **菜單：** 作為旅客，我說「幫我翻譯菜單」→ 眼鏡開相機（實體鍵）→ 拍菜單 → 眼鏡上看到菜名對照、耳邊聽到重點菜色。  
2. **文件：** 作為使用者，我從工具進「文件掃描」→ 拍文件 → 看到原文／譯文卡片，並聽到重點摘要。  
3. **營養（Support→可升 Core）：** 拍餐點 → 看熱量與三大營養 → 確認後寫入飲食 log。

### 4.3 金路徑（Must）

#### GP-U1 菜單翻譯

| 步驟 | 行為 | 驗收 |
|------|------|------|
| 1 | 語音或入口：菜單翻譯 | **不得**導到通用圖片辨識（修現有 Bug） |
| 2 | Action：`app.menu.translate.open`；自動相機模式 | 文案顯示實體鍵提示 |
| 3 | `POST /api/visual/menu-translate` | 1× Vision stream |
| 4 | UI：結構化 JSON 菜名對照；TTS：重點菜色 | `<data>` 雙軌不破版 |
| 5 | 寫 interaction log：`feature_key=menu_translate` | 含 latency、llm_calls=1 |

**LLM SLA：** Vision 1；禁止額外 speech LLM（TTS 用 stream 內文字或模板）。

#### GP-U2 文件掃描 OCR

| 步驟 | 行為 | 驗收 |
|------|------|------|
| 1 | 工具頁 → 文件掃描 | 入口可發現 |
| 2 | `POST /visual/doc-scan` | 1× Vision |
| 3 | UI 原文／譯文卡片 + TTS 重點 | 空圖／模糊有友善錯誤 |
| 4 | log：`feature_key=doc_scan` | 同上 |

#### GP-U3 營養（可選，Support）

- 路徑：`/visual/nutrition` → 使用者確認 → `POST /diet/logs`  
- 存檔成功句：**模板 TTS，0 speech LLM**  
- log：辨識與「確認寫入」分兩事件（採納率）

### 4.4 信任需求

- 拍攝前／中：可見指示 + 語音或 UI「正在拍攝／分析」  
- 不預設背景連拍  
- 對外文案禁止「偷偷幫你看周圍」

### 4.5 不在本航道 v1

- 真比價、匯率自動換算（可列 v1.1）  
- 植物／景點／人物 Wiki（Lab）  
- 收藏系統（Phase B）

### 4.6 對應時序圖

- `時序圖/05-菜單翻譯.png`  
- `時序圖/08-文件掃描OCR.png`  
- `時序圖/06-三餐營養.png`（可選）

## 5. 航道二｜記憶（Hear → Remember）

### 5.1 目標與非目標

| 目標 | 非目標 |
|------|--------|
| 會議可開始錄音、上傳、出摘要與時間軸 | 不做完整錄音筆生態（多說話人完美分離可後做） |
| 語音加行程 → 確認 → 寫入系統曆 | 不做雲端自建行事曆伺服器（v1） |

### 5.2 用戶故事

1. **會議：** 我開會議助理 → 按開始錄音 → 結束上傳 → 看到摘要 Tab + 逐字稿時間軸。  
2. **行事曆：** 我說「幫我安排明天下午兩點開會」→ 看到確認對話框 → 確認後寫入手機曆並 Toast 成功。

### 5.3 金路徑（Must）

#### GP-M1 會議錄音摘要

| 步驟 | 行為 | 驗收 |
|------|------|------|
| 1 | 開啟會議助理 | 頁面可進 |
| 2 | 開始錄音 | **按鈕有反應**（關 P0 Bug） |
| 3 | 結束 → `POST /api/speech/transcribe` | STT + 1 次摘要 LLM |
| 4 | `MeetingResultDialog`：摘要 + 時間軸 | 雙 Tab 可用 |
| 5 | log：`feature_key=meeting_summary` | **speech 路由必須寫 interaction log** |

**LLM SLA：** STT + 1 summary；短會可僅 STT（設定項可後做）。

#### GP-M2 行事曆

| 步驟 | 行為 | 驗收 |
|------|------|------|
| 1 | 語音 → intents → `app.business.calendar.add` | 時間與標題可解析 |
| 2 | `CalendarConfirmDialog` | 可改、可取消 |
| 3 | 確認 → DeviceCalendar 寫入 | Toast 成功 |
| 4 | 埋點 | `calendar_parse` / `calendar_confirm` / `calendar_cancel` 三分開 |

### 5.4 信任需求

- 錄音開始必須有明確 UI／音效狀態  
- 會議場景建議提示「請告知在場者」  
- 取消確認不得寫入曆

### 5.5 不在本航道 v1

- 錄音檔雲端永久庫（可 Phase B）  
- 自動分享到郵件／Notion  

### 5.6 對應時序圖

- `時序圖/09-行事曆.png`  
- 會議：盤點有、圖缺 → **需補時序圖（P1 素材）**

## 6. 航道三｜執行（Intent → Action）

### 6.1 目標與非目標

| 目標 | 非目標 |
|------|--------|
| 一句話完成「意圖→工具→簡短回覆」 | 不做開放域全能 Jarvis |
| ACTION_ONLY 盡量不再呼叫第二段工具 LLM | 不為展示多段 LLM 故意加長鏈 |

### 6.2 用戶故事

1. **天氣：** 「今天台北天氣怎樣」→ 快速得到可聽答（優先模板）。  
2. **喝水：** 「我喝了 300ml」→ 記一筆 → 固定成功句。  
3. **音樂：** 「播放 XXX」→ 打開偏好的音樂 App deep link。

### 6.3 金路徑（Must）

#### GP-E1 意圖主鏈

```
語音/文字 → POST /api/intents/parse
  → ACTION_ONLY：回 actions + reply_text（盡量 1 LLM 或規則）
  → INFO：Dispatcher → 工具 → 表達（模板優先於 speech LLM）
  → Audit log（正規化 feature_key）
```

| 規則 | 說明 |
|------|------|
| 高頻句 | 「天氣」「運勢」「播放」等可走 L0 規則／字典，免 Intent LLM（Phase A 起做天氣） |
| 表達層 | `build_speech_text*` 預設關閉；僅無模板時開啟 |
| 天氣 | 目標：語音路徑 LLM 0–1（現況 3）；直查 0（現況 1） |

#### GP-E2 天氣（Support 但高頻 → 成本樣板）

- 直查：`GET /weather/current` → Open-Meteo → **模板 speech**  
- 語音：Intent slots → 天氣 API → **模板**；拿掉 Weather Parser 第二段 LLM（目標）  
- log：`feature_key=weather`（含直查，不得漏）

#### GP-E3 喝水／簡單記錄

- 寫庫成功 → **固定句 0 LLM**  
- log：`hydration_*`

#### GP-E4 音樂

- Deep link Spotify／YouTube Music／Apple Music  
- 播控（暫停／切歌）不進 v1 Must  

### 6.4 Lab 降級清單

| 功能 | 狀態 | 對外 |
|------|------|------|
| 星座 | Lab；日批後可低成本開 | 不進主宣傳 |
| 商品比價 | Lab；價為 placeholder | 只可說「辨識商品」，不可說比價 |
| 植物／景點／人物 | Lab | Demo 櫥窗 |
| 人物 Wiki + 第二段 speech | Lab | 改模板前不推廣 |

### 6.5 對應時序圖

- `01/02` 天氣、`03` 星座（Lab）  
- 其餘執行類可沿用 intents 總圖（待補架構總圖）

## 7. 平台層需求（三航道共用）

### 7.1 歷程與計費（變現地基）

**事件最小欄位（建議 `feature_usage_event` 或擴充 interaction log）：**

| 欄位 | 說明 |
|------|------|
| `user_id` / `device_id` | 誰、哪台眼鏡 |
| `feature_key` | 見下表 enum |
| `entry` | `voice` \| `ui` \| `system` |
| `status` | `success` \| `fail` \| `cancel` |
| `latency_ms` | 端到端 |
| `llm_calls` | 整條鏈次數 |
| `external_api_calls` | 天氣／Wiki 等 |
| `request_id` | 串 ai_requests |
| `created_at` | 時間 |

**v1 `feature_key` enum（Core + 必要 Support）：**

```
menu_translate, doc_scan, nutrition_scan, diet_log_confirm,
meeting_summary, calendar_parse, calendar_confirm, calendar_cancel,
weather, hydration_log, hydration_query, music_open,
intent_other, auth_*, device_*
```

Lab 可：`horoscope`, `nature_guide`, `sightseeing_guide`, `person_wiki`, `shopping_recognize`（禁止 `shopping_price` 直到真 API）。

### 7.2 成本 SLA（與盤點 §五對齊）

| 場景 | 現況（約） | v1 目標 |
|------|------------|---------|
| 語音天氣 | 3 LLM | 0–1 |
| 直查天氣 | 1 | 0 |
| 菜單／OCR／多數 visual | 1 Vision | 1（+ 圖 hash cache 可後） |
| 飲食／喝水存檔 | +1 speech | 0 |
| 會議 | STT+1 | 維持 |
| 星座 | 2 | Lab；日批後查詢 0 生成 |

### 7.3 信任與 UI 硬規則

1. 所有需相機路徑：實體鍵提示文案統一  
2. 主頁「對話翻譯」不得誤導向圖片辨識（P0）  
3. 播放設定頁背景反白等 UI Bug（P0）  
4. 隱藏／移除 `sdk_test` 與純開發工具入口（對內測包）  

### 7.4 會員（Support 轉換）

- 現況：email + password；改密需舊密碼  
- v1 Must：登入／註冊／refresh 穩定  
- v1 不 Must：Google／Apple、Email 忘記密碼（列轉換優化 backlog）
